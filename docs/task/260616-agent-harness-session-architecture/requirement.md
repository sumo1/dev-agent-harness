# 需求：Agent Harness 会话架构重分层

> 日期：2026-06-16
> 背景：当前产品定位已经从“AI-native task management”转向更明确的 developer-facing agent harness。现有任务、Issue、助理之间的分层开始混在一起，需要先收敛架构，再分步骤改造。

## 原始诉求整理

当前框架是一个 agent harness 工具，但架构分层存在问题：

- 工作区里的 `任务`、`Issue`、`助理`代表三种不同入口：
  - `任务`：复杂任务，适合规划、DAG、子任务和多智能体编排。
  - `Issue`：已知问题，适合直接修复、验证和关闭。
  - `助理`：聊天对话任务，适合开放式协作。
- 三类入口应该有不同界面分类，但共享运行会话能力。
- 顶部的环境变量、工作目录、运行时、自定义选项等，不只是 UI 选择，而是运行时上下文，应该注入模型 prompt，让模型知道当前环境。
- 操作按钮应该是基于当前聊天上下文的快捷指令或运行控制，不应该藏隐式业务逻辑。
- `重试`应该基于当前历史对话追加一条显式指令，而不是绕过上下文重新启动一套隐藏流程。
- `中断`应该是所有运行会话的统一能力，不应该只在某些任务类型里出现。
- 任务类型之间不能耦合。Issue 处理不应该串联到复杂任务里，也不应该以“任务模式”启动。Issue 是明确问题的工程处理，不是规划驱动的大目标。

新增需求：把 OpenClaw 作为一类运行时通道接入。

- 当前已经有 Claude Code、Codex 等运行时，OpenClaw 也应该能作为 runtime provider 被绑定、通讯和展示状态。
- 工作区左侧增加“龙虾”入口，用于查看 OpenClaw 聊天历史、打开分发任务的聊天窗口。
- OpenClaw 里的对话可以分发为当前系统的 `Goal / Issue / Assistant` 会话，但 OpenClaw 本身不是第四种 WorkItem 类型。
- 自动化区增加“龙虾频道”，同步 OpenClaw 已有的定时任务，并允许在本系统内查看、启停、编辑和删除。
- OpenClaw 的历史记录和定时任务优先以 OpenClaw 原生数据为准；本系统只做统一入口、索引和操作转发，不额外维护一套容易漂移的事实副本。

## 需求理解

本轮不是直接改 UI 小细节，而是重定领域模型：

```text
Agent Harness
  = 不同类型的工作入口
  + 统一的 agent session
  + 显式运行上下文
  + 可组合的快捷命令
  + 可插拔 runtime/channel connector
```

不是：

```text
所有入口都塞进 Task
```

## 本任务目标

产出一套可执行的架构改造方案，解决：

1. `任务 / Issue / 助理`的产品语义边界。
2. 统一会话层如何承载三类入口。
3. 环境上下文如何进入 prompt。
4. 重试、中断、继续、验证等按钮如何建模。
5. Issue 如何从复杂任务 DAG 中解耦。
6. OpenClaw 如何作为 runtime/channel/automation source 接入。
7. 后续代码改造的分阶段执行顺序。

## 非目标

- 本任务文档阶段不直接改业务代码。
- 不重新设计 goal DAG 执行引擎。
- 不删除已有数据表。
- 不做大规模 UI 视觉重绘。
- 不把所有内部 `task_queue` 术语一次性改名。内部执行队列可以继续叫 task，但产品层不能继续把所有事情都叫 Task。
- 不在本系统里重建一套 OpenClaw 的完整数据模型。OpenClaw 对话、频道和定时任务的主事实仍然在 OpenClaw。
